# HomieFarms — Coding Instructions

## Project Overview

HomieFarms is an isometric browser farming game. Read [game-design.md](game-design.md) for the full game design document before making any changes.

## Tech Stack

- **Backend:** PHP 8.x with Laravel framework, SQLite database
- **Real-time:** Laravel Reverb (WebSockets)
- **Frontend rendering:** PixiJS 8 loaded via CDN (no npm)
- **Frontend UI:** HTML + Tailwind CSS loaded via CDN (no npm)
- **Font:** Press Start 2P via Google Fonts CDN
- **Auth:** Discord OAuth2 via Laravel Socialite
- **No build tools.** No npm, no Vite, no Webpack, no bundlers. All JS is vanilla `.js` files served directly. All CSS dependencies load via CDN `<script>` or `<link>` tags.

## Architecture Principles

### Server-Authoritative
All game logic runs on the server. The browser client is a rendering-only layer.
- **DO:** Validate every player action server-side before applying it.
- **DO:** Calculate crop growth, weather, economy on the server.
- **DON'T:** Trust any value from the client. Treat client input as untrusted intent.
- **DON'T:** Put game logic in JavaScript. JS only renders and sends user input.

### Data Flow
1. Player interacts (click/tap) → client sends intent to server (REST or WebSocket)
2. Server validates → updates SQLite → broadcasts result via Reverb
3. All connected clients receive update → re-render affected tiles/entities

### File Organization
```
public/js/engine/   — Core rendering: tilemap, camera, input, networking
public/js/entities/  — Visual entities: orb, crops, buildings
public/js/ui/        — UI overlay logic: HUD, panels, chat
public/sprites/      — Exported tile/sprite PNGs (from _resources)
app/Http/Controllers/Api/  — Game API endpoints
app/Services/        — Game logic services (crops, weather, farms)
app/Events/          — WebSocket broadcast events
app/Models/          — Eloquent models
```

## Coding Standards

### PHP (Laravel)
- Follow Laravel conventions (PSR-12, Eloquent, service classes).
- Controllers stay thin — business logic goes in `app/Services/`.
- Use Laravel's built-in validation (`$request->validate(...)`) for all API input.
- Use Eloquent relationships. Don't write raw SQL unless performance requires it.
- Broadcast events for any state change that other players need to see.

### JavaScript (Vanilla)
- No TypeScript, no JSX, no modules requiring bundlers.
- Use ES6+ syntax (classes, arrow functions, const/let, template literals).
- Load scripts via `<script>` tags in order of dependency.
- Each file should define one class or module pattern on a namespace object.
  Example: `window.HF = window.HF || {}; HF.Renderer = class { ... }`
- Keep client code thin — it should only handle rendering and input.

### CSS / Tailwind
- Use Tailwind utility classes for layout and responsive design.
- Custom pixel-art styles go in `public/css/game.css`.
- Do not use `@apply` (requires build tool). Write utility classes inline.
- All UI must work on both desktop (min 1024px) and mobile (min 360px).

## Isometric Grid

- Tiles are diamond/isometric projection.
- Grid uses **axial coordinates** `(q, r)`.
- Screen position from grid: `screenX = (q - r) * tileHalfWidth`, `screenY = (q + r) * tileHalfHeight`.
- Grid position from screen: inverse of the above, accounting for camera offset and zoom.
- Tile sprites from the tileset are roughly 32×32 base.
- Use `image-rendering: pixelated` on the canvas to keep sprites crisp when scaled.

## Pixel UI Rules

- Use Press Start 2P font for ALL text in the game UI.
- Apply `-webkit-font-smoothing: none` and `font-smooth: never` to prevent blurring.
- Pixel borders via CSS box-shadow: `box-shadow: 0 4px 0 0 #000, 4px 0 0 0 #000, ...`
- Color palette from the tileset: browns (#5C3A1E), reds (#8B3A3A), greens (#4A7A3A), purples (#7B6BA5), cream (#F5E6C8).
- Panels use semi-transparent dark backgrounds so the game canvas shows through.
- No gradients, no rounded corners, no smooth transitions. Flat colors, hard edges, snappy animations.
- All touch targets must be at least 44×44px on mobile.

## Multiplayer & WebSocket

- Use Laravel Reverb for WebSocket connections.
- Broadcast on **presence channels** per instance: `instance.{id}`
- Player movement: broadcast throttled to max 10 updates/second.
- Use `.whisper()` for ephemeral events (typing indicators, cursor position).
- Use full broadcast for persistent state changes (crop planted, building built).

## Testing

- Game logic in `app/Services/` should have unit tests (PHPUnit).
- Test crop growth calculations, economy math, zone assignment.
- No frontend tests for MVP — manual testing via browser.
- Test on both desktop Chrome and mobile Safari/Chrome.

## Security

- Discord OAuth2 tokens stored server-side only, never exposed to the client.
- All API endpoints require authentication (Laravel middleware `auth`).
- Rate-limit game actions to prevent spam (Laravel's throttle middleware).
- Validate tile ownership before any farm action (prevent modifying other players' farms).
- Sanitize chat messages server-side before broadcasting.

## Common Gotchas

- The tileset license prohibits redistribution. Exported sprites stay in `public/sprites/` and must not be included in any asset pack or shared separately.
- SQLite has write concurrency limits. Use `BEGIN IMMEDIATE` transactions for write operations.
- PixiJS ticker runs at screen refresh rate. Don't tie game logic to frame rate — use server timestamps.
- Tailwind CDN is fine for development. For production, consider the Tailwind CLI standalone binary (single executable, no npm needed).
- Press Start 2P is hard to read below ~8px. Use minimum `text-[8px]` in Tailwind.

## Git Workflow

- `main` branch is the deployable state.
- Feature branches: `feature/farming-loop`, `feature/multiplayer`, etc.
- Commit messages: concise imperative (`Add crop growth service`, `Fix orb pathfinding`).
- Don't commit `_resources/` source files (`.aseprite`) — only exported PNGs in `public/sprites/`.
